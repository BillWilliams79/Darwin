The four-bucket model

Each bucket loads into a different session type at a different time. Nothing loads twice.

BucketFilesLoaded byWhen
1 · Shared binding rulesCLAUDE.mdEvery session — primary and every workerInjected automatically at session start
2 · Primary auto-memorymemory/MEMORY.md index + memory/*.md topicsPrimary onlyIndex at start (~29 KB budget); topics never auto-load

4 · On-demand topicsAny memory/*.mdAny sessionRead explicitly when the index hook signals relevance
Why workers skip bucket 2 A worker's context is CLAUDE.md (rules) + its requirement (spec) + the code itself. Project history, cost figures, and style preferences are the primary's decision-making load — shipping them to every worker would be pure token cost.
02

Where memories physically live

One tracked home, one visible indirection. Memory is ordinary git-tracked markdown in the DarwinAI-Config repo at <clone>/memory/.

The index contract

MEMORY.md is a pure index — one line per memory, no content bodies. Claude Code loads it at session start under a ~29 KB budget and silently truncates past it, so the index targets under 8 KB. Content lives in topic files, which cost nothing until a session actually reads them.

The symlink

Claude Code looks for auto-memory at ~/.claude/projects/<encoded-path>/memory. A symlink points that location at the tracked <clone>/memory/ — so "AI memory" and "repo docs" are the same files, versioned, reviewed, and merged like any code.

03

The split-brain problem — and the fix

Two machines (macOS + WSL) each run a primary session. Anything that lives on one machine and syncs by hand will diverge — the 2026-07-10 incident proved it.

LayerSync mechanismDivergence bound
CLAUDE.md + memory/ (shared knowledge) DarwinAI-Config SessionStart / throttled pull hook at start; PR merge at closeout one session
Requirements · sessions · swarm state Shared RDS — single source of truth, read via MCP live
User layer (settings.json, statusline, dotfiles) darcfg bare-git-over-$HOME — start-pull + close-push checkpoint (req #2952) one session
Model/effort defaults darwin-canon.json (tracked canon) reconciled into settings.json at session boundaries (req #2963) — a CLI /model switch self-heals, never redefines canon one session
Harness (Claude Code install) Versioned installs (Swarm Independence Program, req #2938/#2939) per-install
The solved state Before req #2952 the user layer synced only when someone remembered to push — split brain by construction. Now every layer either lives in one shared place (RDS) or auto-syncs at session boundaries: pull on start, checkpoint on close. Cross-machine divergence of the primary identity is bounded by one session, and conflicts surface for a human — never auto-clobber.
04

How workers write memory — sync through git

There is no memory-merge machinery. Memory syncs because memory is tracked files.

Durable knowledge — rides the PR

A worker edits <worktree>/memory/<topic>.md with the Edit tool. That lands as a normal commit on feature/<reqId>-<task>; /swarm-complete merges DarwinAI-Config like any other repo, and every other session picks the knowledge up at its next start-pull.

Session-scoped — gitignored, then archived

  • PLAN.md — authoritative copy lands in the MCP session record.
  • memory/SWARM_SESSION.md — archived to ~/.darwin-swarm-archive/ at close-out.
  • Per-worker auto-memory — stays local; the worker promotes what deserves to survive into tracked memory/.
Design tenet Memory is not memorization. Code patterns, file paths, and git history are derivable from the repo — memory files carry only what the code cannot say: preferences, decisions, incidents, and the why behind them.

Groomed presentation. The specs are memory/memory-architecture.md (four-bucket model) and memory/architect-user-repo.md (split-brain end-state, darcfg, canon); the sync scripts live in scripts/swarm/ and are validated by tests/swarm/. When a rule changes, the spec leads and this page is re-groomed.
