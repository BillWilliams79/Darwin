Swarm Branching

When /swarm-start launches a session, it builds that session a private universe: one feature branch replicated across every affected repo, checked out as real git worktrees under one directory. This page states exactly what branches, what deliberately doesn't, and where the seams are.

01

How a session branches

All git state changes route through one skill — /hygiene — invoked once per affected repo. DarwinAI-Config is always first and becomes the worktree root.

Mid-session growth The affected-repo list is a launch-time declaration, not a cage. A worker that discovers it needs another repo invokes /hygiene checkout <repo> — the repo joins the session as a real worktree on the same feature branch, with its own draft PR. Hand-rolled git worktree add is banned; improvised sequences have corrupted repos.
02

Worktree anatomy — real checkouts, no symlinks

The anti-fragile layout (req #2232). Every repo present is a genuine git worktree; repos not in the session are simply absent.

Inside wt-<req>-<task>/

  • Root = DarwinAI-Config worktree — CLAUDE.md, SWARM.md, memory/, scripts/, all checked out on the feature branch.
  • Darwin/, Lambda-*/, DarwinSQL/ — real sub-repo worktrees, only if affected.
  • .swarm-manifest.json — gitignored session state: IDs, branch, PRs, coordination type.

Guard rails

  • Write isolation — a PreToolUse hook blocks edits outside the worktree (plus /tmp and auto-memory paths).
  • No worker writes main — every change flows through the feature branch and its PR; the only direct-to-main path is the primary's closeout skill.
  • Never touch .git/ internals — lock-file trouble stops the session and surfaces to the user (2026-04-14 corruption incident rule).

The design line Git-carried state branches; live shared state does not. The database, the deployed site, and the machine identity are single instances that every session converges on through gates (migrations, deploy locks, session-boundary sync) rather than through branches.

05

What's good — and where the holes are

Strengths

  • True isolation. N sessions edit N disjoint trees — a worker cannot touch a sibling's files, and the write-isolation hook enforces it mechanically.
  • Everything is a PR. Every session leaves a reviewable, revertable GitHub trail from its first minute — including memory and skill changes.
  • Merge-only, commit-only. No rebase anywhere in the machinery; every state change is a commit, and conflicts resolve in-band by the session that owns the work.
  • One branch name across repos ties a multi-repo change into one findable unit — resume rebuilds the whole session from GitHub alone.
  • Anti-fragile layout. Real worktrees, zero symlinks: a corrupted or deleted session directory cannot reach the primary clones.

Holes — known and accepted

  • The database is not branched. Schema DDL and shared-table writes from one session are instantly visible to all. Migration files, darwin_dev-first verification, and RDS snapshots gate it — process, not isolation.
  • Deploy is a shared endpoint. Parallel deployed sessions once raced last-writer-wins on S3; the post-merge deploy + lock (req #2386) serializes it, but deploys still couple sessions in time.
  • Machine-local files sit outside the fence. Concurrent sessions share ~/.claude/settings.json and friends — observed churn under sibling sessions; darcfg surfaces conflicts but can't isolate them.
  • Session-scoped context is thinner than a branch. PLAN.md and session notes are gitignored; they survive only via the MCP record and close-out archive — a crash before archiving loses the notes.
  • Cost scales linearly. Each session is a full checkout (~287 MB node_modules for Darwin) and an independent Claude context — N sessions ≈ N× tokens.
Net assessment The branch fence is drawn exactly around what git can version. Inside it, isolation is total and auditable; outside it (database, deploy, machine identity), the system substitutes gates and session-boundary sync. The holes are all in the second category — shared by design, guarded by process.
