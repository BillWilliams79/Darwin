Version & Branch Rules

The single source of truth for how Darwin numbers every build and decides which branches may be created. Two entities, two version shapes, one reserved-range scheme — nothing more.

Every build is uniquely identified by Major.Minor.Build.Branch — e.g. 1.3.41.2 is Major 1, Minor 3, Build 41, Branch # 2.

01

Two entities — and nothing else

Builds

A point on a line. Each build carries its own major, minor, build_number (B) and branch_number (b), stamped at creation.

No look-back: a build never re-derives its M.m from its branch or neighbours. Its stored tuple is the truth for rendering.

Branches

A line that carries builds. It holds a current M.m used only to stamp new builds, plus a type and a parent_build_fk.

A release is a TYPE of branch, not a separate entity. Customer releases are events on a build (§ colour legend), not a branch type.

The whole model Two version shapes exist: trunk and sub-branch. Which shape a branch uses is decided entirely by its type.
02

Trunk vs. sub-branch

The single distinction that drives every version number.

Trunk  ·  main + release

  • Branch # is always 0.
  • Build # increments monotonically: …8.0 → …9.0 → …10.0.
  • Live M.m line. main's M.m can be reassigned; a release's M.m is fixed for life.

The two trunks never collide — they run different M.m lines (see §3).

Sub-branch  ·  everyone else

  • Build # is FROZEN at the parent build's B at creation.
  • Branch # walks its reserved range (see §4).
  • M.m is a frozen snapshot of the parent.

Types: sample-release (Sprint), csr, hotfix, development, bootleg.

03

Major.Minor is declared, never computed

M.m changes only by management decree — at exactly two moments.

Project creation

The first main branch is seeded with an (M, m) and a First Build # the user supplies in the New-project dialog.

Release hand-off

Creating a release off main carries main's M.m onto the release — the release owns and continues that line. main is left open and must be assigned a new (M, m) before its next build.

No auto-computed new-main There is no default new-main M.m. Creating a release opens main (an out-of-range OPEN_SENTINEL, major/minor = −1) and immediately prompts for the value. Convention bumps the minor (ship 5.0 → main 5.1) — but the engine never assumes it. A build on an open branch is refused until M.m is set.
04

Build # — the B

TypeBuild # behaviour
main (trunk)Increments 1, 2, 3… under the branch's M.m. Resets to 1 (or the seed First Build #) each time management assigns a new M.m after a release.
release (trunk)Continues the trunk sequence at Branch # 0. First build = parent B + 1 (…8.0 → …9.0), then …10.0…. Its M.m never opens, so its B only ever increments.
sample-release · csr · hotfix · development · bootlegFrozen at the parent build's B. Every build on the branch reuses it; only Branch # walks.
05

Branch # — the reserved-range b

Formula shape: base + ord0 × stride + i — where i is the 0-indexed build position on the branch and ord0 is this branch's 0-indexed ordinal among same-type siblings off the same parent build.

TypeFirst build's Branch #StrideSibling-0 range
main / release (trunk)0—0 (always)
sample-release1 + ord0×50 + i501 – 50
csrord1×1000 + i10001000 – 1999

development7000 + ord0×100 + i1007000 – 7099

Decode example …7102 reads as the 3rd build (i=2) of the 2nd development branch (ord0=1) off its parent: 7000 + 1×100 + 2. Stride sets how many builds one sibling can hold before it meets the next sibling's range.

Four principles fix the whole table

  • Development is always available — engineering is king; the whole column is ✓.
  • Bootleg is maintenance — allowed from anywhere, including bootleg-off-bootleg.
  • CSR from any parent — its quality gate is a property of the build (passed a release meeting), not the branch location.
  • Sprint from any parent except another Sprint — "a sprint off a sprint, what a mess."

The two exceptions

  • release is tightly held — created only from main or another release.
  • hotfix may chain off hotfix and off bootleg, but not off development.

Gate is the UI; the database is permissive. The matrix is enforced only in BranchEngine — SQL stores any parent/child pair, so a future "override" UI can bypass it with no schema change. There are currently no creation gates.

Colour legend

Build-dot colours — per build, not per branch

default — normal build green — release ceremony red — hotfix in-flight yellow — bootleg / soak gray — retired / archived

Release-event stars — above the dot, one per event

gold — release / sprint / csr silver — hotfix red — bootleg

A customer release is an event on a build — a star, not a branch type. Star colour is keyed by the branch type it shipped off.
